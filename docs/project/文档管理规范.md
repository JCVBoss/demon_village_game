## 📋 文档 Tag 规范

### Tag 格式

所有文档必须在**第一行**添加如下格式的 tag 注释：

```markdown
<!-- [TAG] 所有者:PO|维护责任人:XXX|创建时间:YYYY-MM-DD|生效标记:✅|最后更新:YYYY-MM-DD|版本:vX.X -->
```

### 字段说明

| 字段 | 说明 | 可选值 |
|------|------|--------|
| **所有者** | 文档最终责任人 | PO（项目所有者） |
| **维护责任人** | 日常维护负责人 | PO/SM/Claude/AliceDesigner/AliceBussiness |
| **创建时间** | 文档首次创建日期 | YYYY-MM-DD |
| **生效标记** | 文档状态 | ✅生效 / ⏳待完善 / ❌废弃 |
| **最后更新** | 最后一次修改日期 | YYYY-MM-DD |
| **版本** | 文档版本号 | v1.0, v1.1, v2.0 等 |

### 维护责任人代码

| 代码 | 角色 | 职责范围 |
|------|------|----------|
| **PO** | Product Owner (JCVBoss) | 需求文档、Backlog、决策文档 |
| **SM** | Scrum Master (AliceDesigner) | 流程文档、协作指南、会议记录 |

## 🔄 文档更新流程

### 1. 修改前

1. 检查文档 tag 中的**最后更新**日期
2. 确认**维护责任人**是否是自己负责的领域
3. 如有必要，在 daily_logs 中记录修改计划

### 2. 修改时

1. 保持 tag 格式在第一行
2. 修改内容后，**必须更新 tag**：
   - 更新 `最后更新:YYYY-MM-DD` 为当前日期
   - 如有重大改动，更新 `版本:vX.X`
3. 提交时使用规范的 commit message：
   ```
   docs(范围): 修改说明
   
   Co-authored-by: 你的名字 <email>
   ```

### 3. 修改后

1. 在 daily_logs 中记录修改内容
2. 如有必要，在协作群中通知相关协作者
3. 推动文档审查（如适用）

## 📊 文档状态标记

### 生效标记说明

| 标记 | 名称 | 说明 |
|------|------|------|
| ✅ | 生效 | 文档已审核通过，可参照执行 |
| ⏳ | 待完善 | 文档草稿中，内容可能不完整 |
| ❌ | 废弃 | 文档已过期，不再参照 |

### 版本命名规则

- **v1.0** - 初始版本
- **v1.1, v1.2** - 小幅修改（错别字、格式调整）
- **v2.0** - 重大更新（结构调整、内容重写）

## 📝 示例

### 正确的 tag 示例

```markdown
<!-- [TAG] 所有者:PO|维护责任人:AliceDesigner|创建时间:2026-03-16|生效标记:✅|最后更新:2026-04-03|版本:v1.0 -->

# 角色设定 - 10 位村民
```

### 错误的 tag 示例 ❌

```markdown
<!-- 创建者：Alice，2026-03-16 -->  ← 格式不规范

# 角色设定
```

```markdown
[所有者：PO，维护：SM]  ← 不是 HTML 注释格式

# 角色设定
```

## 🔧 自动化脚本

### 批量添加 tag（Linux/Mac）

```bash
# 为目录下所有 .md 文件添加 tag
cd docs/design
for f in *.md; do
  if ! head -1 "$f" | grep -q "\[TAG\]"; then
    sed -i '1i<!-- [TAG] 所有者:PO|维护责任人:AliceDesigner|创建时间:2026-04-03|生效标记:✅|最后更新:2026-04-03|版本:v1.0 -->' "$f"
    echo "Updated: $f"
  fi
done
```

### 检查缺失 tag

```bash
# 查找没有 tag 的 markdown 文件
find docs -name "*.md" -exec sh -c '! head -1 "$1" | grep -q "\[TAG\]" && echo "Missing TAG: $1"' _ {} \;
```
