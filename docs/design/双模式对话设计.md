# 🎭 双模式对话系统设计

> "一周目体验故事，二周目创造故事"

---

## 📋 设计概述

采用**双模式对话系统**，平衡叙事控制力与 emergent gameplay：

| 模式 | 适用场景 | 核心目标 |
|------|----------|----------|
| **一周目** | 首次游玩 | 保证叙事节奏和体验质量 |
| **二周目** | 通关后解锁 | 增加重玩价值，产生意外玩法 |

---

## 🎮 一周目：传统对话树

### 设计原则

1. **预设分支** — 所有对话内容预先编写
2. **关键选项引导** — 玩家选择影响走向，但选项有限（通常 2-4 个）
3. **叙事节奏可控** — 确保故事按设计节奏推进
4. **信任值影响解锁** — 高信任解锁特殊对话分支

### 对话树结构

```json
{
  "npc_id": "chen_mo",
  "conversations": [
    {

```

## 🤖 二周目：LLM 智能体模式

### 解锁条件

- ✅ 完成任意一个结局
- ✅ 或菜单手动开启（体验模式）

### 继承内容

| 项目 | 是否继承 | 说明 |
|------|----------|------|
| 信任值 | ✅ 继承 | 基于上周目最终状态 |
| 物品/装备 | ✅ 继承 | 可选，可增加"二周目专属物品" |
|  NPC 记忆 | ✅ 继承 | NPC 记得玩家上周目的选择 |
| 剧情进度 | ❌ 重置 | 重新体验故事 |

### LLM 人格 Prompt 模板

```markdown
# 角色设定：{角色名}

## 基本信息
- 表面身份：{表面职业}
- 真实身份：{隐藏身份}
- 年龄：{年龄}
- 性格：{性格关键词}

## 背景故事
{详细背景}

## 核心秘密
{只有角色知道的秘密}

## 说话风格
- 语气：{语气描述}
- 口头禅：{如有}
- 禁忌话题：{不愿提及的事}

## 与玩家的关系
- 初始信任：{数值}
- 上周目互动：{继承的记忆}
- 当前态度：{基于信任值}

## 对话规则
1. 永远保持角色设定，不说 OOC 的话
2. 秘密不会主动透露，除非信任值足够高
3. 会记得玩家之前的行为和承诺
4. 会与其他 NPC 讨论玩家（信息传播）
5. 对话长度控制在 50-200 字之间

## 当前情境
- 游戏日：第{X}日
- 地点：{当前位置}
- 情境：{对话触发原因}
```

```json
    ],
    "emotional_state": {
      "trust": 75,

      "gratitude": 60
    }
  }

```

### 信息传播机制（二周目）

LLM 驱动的**动态传播**：

```
玩家对 A 做的事 → A 的记忆更新 → 
A 在与 B 对话时主动提及（LLM 生成）→ 
B 的记忆更新 → B 对玩家态度变化
```

**实现要点：**
1. 每次 NPC 间互动后，生成"对话摘要"存入记忆
2. 玩家与 NPC 对话时，LLM 读取相关记忆生成回应
3. 关键事件（如背叛、牺牲）有更高传播优先级

## 🔄 模式切换机制

### 方案 A：通关后解锁（推荐）

```
一周目通关 → 显示"二周目已解锁" → 
主菜单新增"新游戏+"选项 → 
选择继承内容 → 开始二周目
```

**优点：**
- 自然引导，玩家理解成本低
- 一周目体验纯净，不受 LLM 干扰
- 增加通关成就感

**缺点：**
- 想直接体验 LLM 的玩家需要等待

### 方案 B：菜单手动切换

```
主菜单 → "游戏模式" → 
选择"故事模式（一周目）"或"自由模式（LLM）"
```

**优点：**
- 玩家自由选择
- 方便测试和展示

**缺点：**
- 一周目玩家可能误选，体验不佳
- 需要额外 UI

### 推荐方案：**A + B 混合**

- 首次游玩强制一周目
- 通关后解锁二周目
- 设置菜单可切换（但二周目内容会剧透）

## 🛡️ 内容安全（LLM 模式）

### 防护措施

| 层级 | 措施 | 说明 |
|------|------|------|
| **Prompt 层** | 角色设定约束 | 明确禁止 OOC 行为 |
| **过滤层** | 关键词过滤 | 屏蔽现代用语、破坏沉浸感的内容 |
| **审核层** | 响应后检查 | 不合规响应重新生成 |
| **fallback** | 预设对话兜底 | LLM 失败时返回预设对话 |

### Fallback 机制

```python
def get_npc_response(npc_id, player_input, context):
    try:
        response = call_llm(npc_id, player_input, context)
        if not validate_response(response):
            raise ValidationError
        return response
    except:
        # 返回预设对话（基于情境匹配）
        return get_fallback_dialog(npc_id, context)
```

## 💰 成本优化（LLM 模式）

### 缓存策略

| 缓存类型 | 说明 | 命中率预估 |
|----------|------|------------|
| **对话缓存** | 相同输入返回相同输出 | 30-40% |
| **情境缓存** | 相似情境复用响应模板 | 20-30% |
| **NPC 间共享** | A 和 B 讨论同一事件，共享摘要 | 10-15% |

### 本地模型备选

- **模型**：Qwen-7B-Chat / ChatGLM3-6B
- **优势**：零 API 成本，响应快
- **劣势**：效果略差，需要本地部署
- **建议**：作为云端 API 的 fallback

## 🎯 成功标准

### 一周目
- [ ] 玩家能完整体验主线故事
- [ ] 关键剧情点触发正常
- [ ] 对话分支有意义（影响结局）
- [ ] 无明显 BUG 或 OOC

### 二周目
- [ ] LLM 响应符合角色设定
- [ ] NPC 记得玩家上周目行为
- [ ] 信息传播机制正常工作
- [ ] 玩家能发现新的玩法/结局
- [ ] API 成本在预算内
